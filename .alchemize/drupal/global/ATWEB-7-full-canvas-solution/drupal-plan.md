# ATWEB-7: Full Canvas Solution — Implementation Plan (Iteration 7)

## Ticket Analysis

- **Core request**: Enable Canvas's "template + per-content slots" hybrid model for node entities (e.g. Article). A ContentTemplate defines the default layout with exposed slots, and each node stores its own component tree for those slots. Currently this works in tests but is blocked in production by multiple gates.
- **Key constraint**: The ticket explicitly states "we WILL be editing contrib module canvas directly and will contribute the work back." Changes go directly into `web/modules/contrib/canvas/` source files — NOT patches.
- **Affected areas**: Canvas contrib module (`web/modules/contrib/canvas/`) — specifically the component tree loader, access checker, API layout controller (GET/POST/PATCH), client data converter, content template entity, API content controllers, routing, boot controller, and exposed slot API surface.

Also added to Step 1's `hasContentTemplateWithExposedSlots()` on `ComponentTreeLoader` for the loader context. **Reviewer concern**: Step 9 identified the `{node}` → `{entity}` parameter mapping problem but left multiple alternatives open instead of committing to one.

**Resolution**: Step 9 now commits to a **dedicated route** approach: a new route `canvas.node.layout` with path `/node/{node}/layout` and a controller method that redirects to the `canvas.boot.entity` route with the correct parameters (`entity_type: 'node'`, `entity: $node`). This is cleaner than a deriver (no dynamic task generation needed — the tab visibility is controlled by the access check on the new route) and avoids the parameter mapping complexity of trying to use `canvas.boot.entity` directly as the local task route. ## Architectural Decision: Frontend Save Data Contract

**Decision**: The frontend sends the **full merged tree** (template + slot content) on POST save. The backend extracts the slot subtree before persisting to the node's Canvas field.

**Rationale**:
1. The Canvas editor already works with the merged tree internally — components in the editor are not tagged with "slot" vs "template" origin at the component level in the tree structure.
2. Requiring the frontend to extract the slot subtree before sending would duplicate slot boundary logic on both frontend and backend, creating a divergence risk.
3. The existing `post()` → `updateEntity()` flow receives the full `content` region layout from the frontend. The slot extraction is a backend-only concern, keeping the frontend contract simple: "send the tree as you see it."
4. `SlotTreeExtractor.extractSlotSubtree()` performs the extraction in `updateEntity()` before calling `convert()` on the node's Canvas field.
5. `SlotTreeExtractor.isSlotComponent()` remains necessary for `patch()` (Step 6b) to determine if a component being auto-saved belongs to the template (reject with 403) or the slot (allow and auto-save to the node).

**Impact on steps**:
- **Step 6 (POST)**: `updateEntity()` receives the full merged tree, calls `SlotTreeExtractor.extractSlotSubtree()` to filter down to slot components only, then passes the filtered subtree to `convert()`. Model data filtering is handled implicitly — `convertClientToServer()` iterates layout components, not model keys, so template-component model entries are naturally excluded.
- **Step 6b (PATCH)**: `patch()` uses `getPerContentTemplate()` + template tree UUID lookup to determine if the edited component is template-owned (reject) or slot-owned (allow). Since the entity's Canvas field only contains slot components, `getEntityWithComponentInstance()` naturally finds slot components in the node's tree.
- **Step 7 (GET)**: `get()` returns the full merged tree with `editable` metadata per component. The frontend uses this merged tree for editing and sends it back on POST.
- **Step 7b (preview + PATCH response)**: `buildPreviewRenderable()` and `buildLayoutAndModel()` use the merged tree for accurate preview and PATCH response rendering.

### Step 6: Make the POST save path slot-aware for per-content editing

- **Type:** Code Change (direct contrib edit)
- **Script:** N/A
- **drupal.org issue:** [#3498525](https://www.drupal.org/project/canvas/issues/3498525)
- **Files:**
  - `web/modules/contrib/canvas/src/Controller/ApiLayoutController.php` — `updateEntity()` (lines 556–579)
  - `web/modules/contrib/canvas/src/ClientDataToEntityConverter.php` — `convert()` (lines 48–62)
  - **New service:** `web/modules/contrib/canvas/src/Storage/SlotTreeExtractor.php`
- **Reference docs:** `canvas-content-templates.md` → "Exposed Slots and Per-Content Editing"
- **Details:**
  1. **The problem**: When saving a node's layout via POST, `updateEntity()` delegates to `ClientDataToEntityConverter::convert()`, which writes the **full** component tree to the entity's canvas field. For per-content editing with exposed slots, the save path receives the **full merged tree** (template + entity slot content) from the frontend (see Architectural Decision above). It must:
     - Extract **only** the subtree belonging to exposed slots
     - Persist that subtree to the node's Canvas field
     - Leave the template's tree untouched
  2. **New service: `SlotTreeExtractor`** (create at `web/modules/contrib/canvas/src/Storage/SlotTreeExtractor.php`):
     - This logic is needed in multiple places (POST save here, PATCH guard in Step 6b, GET merge verification), so it warrants a dedicated service rather than a method on `ApiLayoutController`.
     - Add to `canvas.services.yml` with injection of `ContentTemplate` entity storage.
     - Method: `extractSlotSubtree(array $merged_tree_components, ContentTemplate $template): array` — given the merged tree's components array and the ContentTemplate, returns only the components belonging to exposed slots. Uses the template's tree as the reference: any component NOT in the template's tree is a slot component.
     - Method: `isSlotComponent(string $component_uuid, ContentTemplate $template): bool` — determines if a specific component (by UUID) belongs to the template tree or is a slot component. Checks if the UUID exists in `$template->getComponentTree()`. If it exists in the template → it's a template component (returns false). If it doesn't → it's a slot component (returns true).
     - **Leveraging existing infrastructure**: The `ComponentTreeItemList::componentTreeItemsIterator()` with `isChildOfComponentTreeItemSlot()` predicate (used in `injectSubTreeItemList()` at line 540) already provides tree filtering logic. `SlotTreeExtractor` should use this pattern — comparing the entity's tree component UUIDs against the template's tree component UUIDs.
  3. **Tree extraction algorithm** (implemented in `SlotTreeExtractor::extractSlotSubtree()`):
     ```
     Given merged_tree_components[] and template:
       template_uuids = collect all UUIDs from template.getComponentTree()
       slot_components = filter merged_tree_components where UUID NOT IN template_uuids
       Return slot_components (these are the per-entity slot content)
     ```
     This is simpler and more robust than walking exposed_slot definitions recursively because:
     - Template components have stable UUIDs (stored in config)
     - Slot components have per-entity UUIDs (never in the template's tree)
     - The approach is O(n) with a UUID set lookup
  4. **In `ApiLayoutController::updateEntity()`**:
     - When the entity is a `FieldableEntityInterface` (node), use `getPerContentTemplate()` to check if per-content editing mode applies
     - If yes: use `SlotTreeExtractor::extractSlotSubtree()` to extract only the slot subtree from the merged layout's `components` array before passing to `convert()`
     - The extracted subtree replaces the `components` in the layout passed to `convert()`. The layout structure (`nodeType: 'region', id: 'content'`) is preserved — only the `components` array within the region is filtered to contain only slot components
  5. **Model data filtering — implicitly handled, no separate extraction needed**: `ClientDataToEntityConverter::convert()` at line 61 calls `self::convertClientToServer($layout['components'], $model, ...)` which iterates the `$layout['components']` array — **not** the `$model` keys. When the layout components array has been filtered to slot-only components by `SlotTreeExtractor::extractSlotSubtree()`, `convertClientToServer()` only processes those slot components and looks up their corresponding `$model` entries by UUID. Template-component UUIDs are no longer in the filtered `$layout['components']` array, so their model entries in `$model` are never accessed or written to the entity. **No separate model filtering step is needed** — the model data is filtered implicitly by the layout component filtering.
  6. **In `ClientDataToEntityConverter::convert()`**:
     - The `load()` call at line 55 (`$item_list = $this->componentTreeLoader->load($entity)`) loads the node's Canvas field item list. After Step 1's loader relaxation, this works for nodes with ContentTemplate + exposed slots. The returned item list is the node's existing slot tree (or empty for a new node).
     - The `setValue()` call (line 61) on the component tree field should work with the filtered subtree — each item retains its `parent_uuid` and `slot` pointing to the template's component UUID
     - The assertions at lines 57–59 (`$layout['nodeType'] === 'region'`, `$layout['id'] === 'content'`) remain valid because the slot subtree is still wrapped in the region structure
     - No changes needed to `convert()` itself — the filtering happens before the call in `updateEntity()`
- **Expected outcome:** When saving a node's layout via the Canvas editor POST, only the per-entity slot content is persisted to the node's Canvas field. The template tree is never modified by per-content saves. Model data is filtered implicitly through the layout component filtering.
- **Depends on:** Steps 1, 4, 5

     canvas.node.layout:

         _title: 'Layout'
       requirements:
         _custom_access: 'Drupal\canvas\Access\NodeLayoutAccessCheck::access'

     public function nodeLayout(NodeInterface $node): HtmlResponse {

The route uses `_custom_access: 'Drupal\canvas\Access\NodeLayoutAccessCheck::access'` which resolves the class directly from the container via the FQCN reference — no tag-based `applies_to` lookup is used for `_custom_access` routes. ## Risks & Dependencies

| Risk | Impact | Mitigation |
|------|--------|------------|
| **Breaking existing `canvas_page` functionality** | High — canvas_page is the primary page building entity type | Keep all canvas_page code paths unchanged. The restriction relaxation only adds new code paths for eligible entities. Run full Canvas test suite after each change. `getPerContentTemplate()` returns null for canvas_page entities (they implement `ComponentTreeEntityInterface`). |
| **Frontend (React app) not handling template context** | Medium — backend changes are useless without frontend awareness | Steps 4, 7, 7b, 8 provide the data contract. Frontend work items explicitly documented above. The team knows what remains after this plan is implemented. |
| **Multiple exposed slots not supported** | Low — ticket and ADRs scope to single exposed slot | The code asserts `count(getExposedSlots()) === 1` (ContentTemplate::build() line 365). Keep this assertion in `getMergedComponentTree()`. Issue #3526189 must be resolved before supporting multiple slots. |
| **ComponentTreeLoader change affects other callers** | Medium — the loader is used by access checker, view builder, and API controllers | All callers mapped in Step 1 (ComponentTreeInputExtractor, ComponentInstanceForm, GeneratedFieldExplicitInputUxComponentSourceBase). Read-only consumers work transparently. ComponentInstanceForm loads the node's slot tree (not merged) — correct for instance editing. |
| **Exposed slot validation hardcoded to `full` view mode** | Low — MVP only targets full view mode | No change needed for MVP. The `ValidExposedSlot: full` constraint correctly limits to full view mode. |
| **Save path tree extraction correctness** | High — incorrect extraction corrupts entity data | The extraction algorithm uses template UUID comparison (O(n) set lookup) — simpler and more robust than recursive tree walking. Thoroughly tested with edge cases in Step 12. `SlotTreeExtractor` service enables focused unit testing. Model data is implicitly filtered by `convertClientToServer()` iterating layout components — no separate model extraction step needed. |
| **drupal.org maintainer alignment** | Medium — changes may conflict with maintainers' vision | Engage in issues #3498525 and #3518248 before starting implementation. Share the plan. Start with WS1 (most clearly needed) and iterate. |
| **Editing contrib directly** | Medium — changes overwritten by `composer update` | Ticket explicitly requires this approach. Document all changes clearly for contribution back to drupal.org. Consider creating patches as a parallel track for safety. |
| **PATCH auto-save for slot editing** | High — broken auto-save degrades UX significantly | Step 6b explicitly addresses this with correct ordering: template tree check BEFORE `getEntityWithComponentInstance()`. Template-owned components are rejected with 403. Slot components auto-save to the node. |
| **Create links for nodes before route generalization** | Medium — broken UI links | Step 10 depends on Step 3. `getAllContentEntityCreateLinks()` auto-discovers node:article after the field is added. Route generalization (including bundle in create link attributes) must be complete first. Dependency order enforced. |
| **Auto-save data format for slot editing** | Medium — corrupted auto-save breaks the editor | Step 6c explicitly addresses auto-save round-trips. Auto-save stores the node's Canvas field (slot subtree only). The merge with the template tree happens fresh on each `get()` after auto-save restoration using `getMergedComponentTree()`. Tests must verify round-trip correctness. |
| **Disabled ContentTemplate with exposed slots** | Low — edge case, now fully addressed | `getPerContentTemplate()` checks `$template->status()` — a disabled template returns null. `hasContentTemplateWithExposedSlots()` on `ComponentTreeLoader` also checks `$template->status()`. `getTemplateContext()` on `CanvasController` checks `$template->status()`. All three are consistent. Edge case test in Step 12 item 10 verifies this end-to-end. |
| **`getMergedComponentTree()` and `build()` divergence** | Low — mitigated by design | `build()` now delegates to `getMergedComponentTree()` → `toRenderable()`. There is only one merge implementation. Any change to merge logic automatically applies to both rendering and editor loading. |
| **Existing ContentTemplate conflict** | Low — addressed in Step 11 | Step 11 explicitly loads and updates the existing `node.article.full` template instead of creating a new one. The script adds a Wrapper root, re-parents existing components, and sets exposed_slots. No config conflict. |
| **Preview rendering broken for per-content editing** | ~~High~~ Addressed | Step 7b fixes `buildPreviewRenderable()` to use `getMergedComponentTree()` for nodes in per-content mode. All three callers (`get()`, `patch()`, `post()`) produce correct previews including the template shell. |
| **PATCH response layout incomplete for per-content editing** | ~~High~~ Addressed | Step 7b fixes `buildLayoutAndModel()` to use merged tree with editable metadata for PATCH responses. The frontend receives consistent layout data whether loading via GET or receiving PATCH updates. |
| **Per-content editing detection duplication** | ~~Medium~~ Addressed | `getPerContentTemplate()` centralizes all conditions (FieldableEntityInterface, !ComponentTreeEntityInterface, template exists, status(), exposed slots) in one helper method. Used by `get()`, `patch()`, `buildPreviewRenderable()`, `buildLayoutAndModel()`, and `updateEntity()`. |
| **Delete route backward compatibility** | Low — addressed in Step 3 | Frontend obtains delete URLs from operation links via `getEntityOperations()` → `getUrlFromRoute()`. The `match()` closure already handles `entity_type` and `entity` parameter names (lines 428-429). No frontend changes needed for delete URL construction. |
